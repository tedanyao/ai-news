Agent Workflow Improvements

1. Make Permissions Match The Task Shape

Claude Code 2.1.178 added permission rules that can match tool input parameters, such as blocking Agent(model:opus) while allowing other agent use. This is exactly the direction agent safety needs: not just "allow Bash" or "deny Read," but narrower rules based on what the tool is being asked to do.

For your own workflow, think in policy categories:

  • Read-only exploration
  • Local file edits
  • Dependency changes
  • Network access
  • Subagent spawning
  • MCP calls
  • Credential-adjacent file reads
  • Production or cluster commands

The useful question before each task:

Which capability does this task actually need, and which capabilities should remain blocked?

2. Audit Subagents Like Background Jobs

The recent Claude Code fixes around subagent transcripts, backgrounding, blocked sessions, and stale "Working" states are a reminder that subagents are not magic helpers. They are background jobs that need observability.

For multi-agent work, require:

  • One named goal per subagent
  • Explicit allowed files/tools
  • A transcript or summary you can inspect
  • A stop condition
  • A final artifact or evidence bundle
  • No silent escalation to broader scope

If a subagent cannot explain what it did, it should not be trusted more than an unlogged shell script.

3. Choose The Agent's Runtime And Language Deliberately

The polyglot chess-engine study is useful because it says a subtle thing: agents can produce working code in many languages, but language choice still changes the artifact.

For backend/platform tasks:

  • Use the repo's native language for production changes
  • Use shell/Python/Go for operational scripts only when they fit team maintenance norms
  • Avoid exotic languages for generated tooling unless the goal is exploration
  • Ask for performance and maintainability tradeoffs before accepting a generated implementation
  • Require stronger validation when the agent works outside the repo's familiar language ecosystem

The agent may be polyglot, but your team still owns the code.

Application-Side Practices

Treat Agent Sessions As Job Records

The practical daily habit is to stop thinking of agent work as a chat and start thinking of it as a job record.

Every meaningful session should have:

  • Job name
  • Scope
  • Policy mode
  • Runtime/worktree
  • Tools used
  • Subagents launched
  • Commands run
  • Validation evidence
  • Final state
  • Follow-ups

This maps naturally to platform engineering because it looks like any other background worker or CI job.

Use "Safe Mode" As A Debugging Pattern

Claude Code recently added safe mode to start with customizations disabled. Even if you are not using that exact feature, the pattern is valuable.

When an agent behaves strangely, rerun the task with:

  • No custom instructions
  • No MCP servers
  • No plugins
  • No skills
  • Read-only mode
  • Minimal context

If the behavior changes, your issue may be configuration, not model capability.

Programming Languages Still Matter

The chess-engine study found frontier agents can build non-trivial systems across many languages, including unusual targets, but language choice still affects performance, cost, feature choices, and validation burden.

Why it matters: do not let an agent pick a language or dependency ecosystem casually. The artifact has to be maintained by humans.

Applied Tip

Add this to your agent instructions for generated tools:

If you propose creating a new script or tool:
- use the repository's existing language/tooling unless there is a strong reason not to
- explain language/runtime tradeoffs
- avoid new dependencies without approval
- include the exact command to run it
- include a cleanup/removal path if it is temporary

This prevents the common "helpful throwaway script becomes permanent weird tool" problem.
