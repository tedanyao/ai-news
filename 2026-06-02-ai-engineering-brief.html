Agent Workflow Improvements

1. Add A Critic Pass As A Default Step

GitHub's rubber duck feature is a useful signal because it turns "ask for a second opinion" into a first-class workflow primitive. You do not need Copilot CLI to adopt the pattern.

For any non-trivial task, add this review step:

Before continuing, review your current plan/design/tests as a critic.
Look for:
- hidden assumptions
- scope creep
- missing tests
- risky commands
- production impact
- simpler alternatives

Return concrete feedback and revise the plan only after the critique.

This is especially useful before agents edit infrastructure, auth, migrations, shared libraries, CI, or Kubernetes manifests.

2. Schedule Agent Checks, Not Just Agent Work

The new /every and /after commands in Copilot CLI are interesting because they make recurring checks part of an active session. The obvious use is "run tests every 30 minutes," but the better pattern is scheduling evidence collection.

Useful scheduled prompts:

  • Every 30 minutes: summarize what changed and what evidence exists
  • Every hour: report token/credit usage and stop if the task is drifting
  • After 20 minutes: ask whether the current approach still matches the original scope
  • After a test run: summarize failures by likely root cause, not raw output

This turns the agent from a long-running black box into something that periodically produces reviewable state.

3. Use Sandboxes As A Workflow Boundary

GitHub's cloud and local sandbox preview matters for platform engineers. Sandboxing is not just security theater; it is what makes autonomous tool execution usable.

Practical distinction:

  • Local sandbox: good for repo work that needs your local files but should have limited filesystem/network access
  • Cloud sandbox: good for parallel experiments, reproducible throwaway work, and offloading compute without touching your laptop state

The important habit is to choose the runtime before starting the task. If the task is exploratory, risky, or potentially messy, put it in a sandbox or worktree first.

Application-Side Practices

Agent Canvases Are A Hint About The Future

GitHub describes canvases as shared work surfaces where the agent updates a structured artifact and the human can inspect, edit, reorder, approve, or redirect it. That is the right mental model even outside Copilot.

You can simulate a canvas today with Markdown:

# Agent Canvas

## Goal

## Current Plan

## Evidence Collected

## Decisions Needed

## Diff Summary

## Validation

## Risks

## Next Action

Ask the agent to keep this updated during longer work. It is better than relying on a scrolling transcript because the important state has a fixed place to live.

Cost Is Now Part Of Agent Workflow Design

GitHub's move to usage-based Copilot billing started June 1. The practical point is not just pricing; it is that long-running agent sessions now have an operational cost shape. Large context, long outputs, cached tokens, code review, and cloud sandbox compute can all matter.

Cost-aware agent habits:

  • Use narrow prompts and specific repo context
  • Ask for file maps before dumping huge files
  • Prefer targeted rg searches over broad "read the whole repo"
  • Stop or summarize long sessions before they drift
  • Use cheaper/fast models for exploration when possible
  • Save reusable findings in WORKLOG.md or AGENTS.md
  • Treat scheduled prompts as useful only when they produce real evidence

This is not about being cheap. It is about keeping agent work intentional.

Daily Workflow Upgrade

Create a "session heartbeat" prompt for any agent task expected to run more than 15 minutes.

Every time you complete a meaningful step, update this heartbeat:

Goal:
Current hypothesis:
Files inspected:
Files changed:
Validation run:
Evidence:
Open risks:
Next action:
Stop condition:

If the next action expands scope, stop and ask.

This is the lightweight version of canvases, scheduled prompts, and session observability. It makes the agent's work inspectable without needing a new tool.

Tool Or Release Watch

GitHub Copilot CLI: Rubber Duck, Scheduling, Voice

GitHub Copilot CLI's Build refresh added a built-in critic agent, prompt scheduling with /every and /after, local voice input, and an experimental terminal UI with repository tabs.

Why it matters: these are workflow primitives, not just UI features. Critique, recurring checks, and better issue/PR context are exactly what agent work needs to become reviewable.

GitHub Copilot App: Canvases And Shared Work State

The Copilot app technical preview is now available to more Copilot customers and adds canvases, cloud sessions, cloud automations, app/CLI session sharing, agentic browsing, and /chronicle for querying prior agent session data.

Why it matters: agent UX is moving from "chat transcript" to "visible work state." That is a better fit for software engineering, where plans, diffs, tests, and risk need to stay inspectable.

GitHub Copilot Sandboxes

Cloud and local sandboxes for Copilot are now in public preview. Local sandboxes restrict filesystem/network/system access on your machine. Cloud sandboxes run in isolated GitHub-hosted Linux environments and can be resumed across devices.

Why it matters: agent adoption in real teams needs execution isolation, policy, and lifecycle control. This is especially relevant for platform and infra workflows.

Copilot SDK GA

The Copilot SDK is now generally available across Node/TypeScript, Python, Go, .NET, Rust, and Java. It exposes planning, tool invocation, file edits, streaming, multi-turn sessions, MCP/custom tools, prompt customization, tracing, auth options, cloud/remote sessions, and hooks.

Why it matters: internal developer tools can embed agent workflows instead of shelling out to a chat UI. For platform teams, this points toward internal agents with real tracing, permissions, and tool governance.

Codex And Claude Code Runtime Details

Codex 0.135.0 added richer diagnostics, remote connection status, named permission profiles, and sandbox presets in the Python SDK. Claude Code's recent changelog focuses on long-session memory/performance, background agent control, permission prompt clarity, and concurrent-agent fixes.

Why it matters: the "boring" runtime details are where agent productivity lives.

Applied Tip

Before launching a long-running agent task, classify it:

Runtime:
- local unsandboxed
- local sandboxed
- cloud sandbox
- separate worktree
- no-edit investigation

Review:
- critic pass required?
- scheduled heartbeat required?
- final diff review required?

Cost:
- expected session length
- expected model/tool usage
- stop if scope expands?

This makes you choose the right operating mode before the agent starts spending time, tokens, or risk budget.

Workflow To Try

Run one agent task today with a heartbeat and critic pass.

Prompt:

Use a session heartbeat while working.
Before editing, fill in Goal, Current hypothesis, Files to inspect, Validation, Risks, and Stop condition.
After your initial plan, run a critic pass against the plan.
Then proceed only if the plan is still within scope.
After work, update the heartbeat with files changed, validation evidence, and remaining risks.

Use it on something small:

  • PR description cleanup
  • one failing test
  • one docs/runbook update
  • one CI failure investigation
  • one narrow refactor

The point is to practice observable delegation.
